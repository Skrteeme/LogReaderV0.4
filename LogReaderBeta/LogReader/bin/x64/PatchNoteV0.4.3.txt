1) Произведён рефакторинг кода с целью ускорения его работы. Теперь при загрузки архива с логами, создаётся БД для этого РВ и заполняется информацией из логов
   и далее поиск осуществляет по БД, а не по логам. БД нигде не сохраняются, кроме временной памяти и удаляютя при передаче следующего архива с логами,
   либо выхода из программы.
2) TelegramClient был перенесён в отдельное пространство имён от UI, чтобы избежать наложения БД.
3) Изменён алгоритм работы TelegramClient:
	a) Теперь работа TelegramClient осуществляется с каждым пользователем индивидуально по ID пользователя.
	  На каждого пользователя выделяется своя область памяти, со своей базой. Для того чтобы при одновременном
	  запросе от нескольких пользователей, анализ производился в той порции логов, которую передал именно этот пользователь.
	б) База, по загруженному архиву с логами, сохраняется в памяти до загрузки другого архива(в этом случае перезаписывается), либо до закрытия утилиты.
	   Таким образом, в случае, если список SGTIN превышает допустимое Telegram значение на размер одного сообщения, то появляется возможность обработки
	   нескольких сообщений в одной базе сразу(Важно: если SGTIN был резделён пополам в 2 сообщениях, то он обрабатываться не будет). В этом случае, будет
	   получена порция ответов соответствующая кол-ву отправленных запросов с искомыми SGTIN.
	в) Изменён вывод ответа по анализу логов. Теперь если SGTIN были найдены и эти отчёты были успешно отправлены в ИС МДЛП, то будет выводится краткая
	  информация об отчёте(номер, дата, мод). Если же, процесс выбытия был нарушен или что то пошло не так (на данный момент обрабатываются исключения:
	  "МБ не инициализирован", "Связь со спутником отсутствует более 24 часов", "Bad file descriptor", отсутствие ответа от МБ о принятии отчёта), то будет выводится подробная информация об отчёте, РВ, состоянии МБ 
	  и СБ на момент выбытия. В самом низу отчёта будут выводится SGTIN, информация о выбытии которых отсутствует(если таковые есть).  
	г) Теперь в ответ на запрос, будет приходить 2 файла. Один с анализом выбытия нужных SGTIN, второй с ошибками МБ, СБ, модуля ГНСС и информацией по питанию
   	  за время логов обмена СБ МБ(до 14 дней).
	д) Добавлена функция генерации пароля тех.поддержки по вводу 2 последних цифр номера РВ.
	е) Добавлено скачивание, распаковка архивов в индивидуальные папки для каждого ID, чтобы в случае одновременной загрузки небыло коллизий.
	ж) Добавлены поэтапные проверки состояний процессов. Необходимость при нестабильном интернете, для синхронизации и корректной работы процессов.
4) Добавлен анализ логов по ошибкам МБ, СБ, ГНСС модуля и информации по питанию.
5) Добавлен анализ места работы устройства по координатам и вывод адреса непосредственного места осуществления выбытия. Информация адресной книги по API берётся
   из базы ФИАС, поэтому возможно некорректное отображение адреса, так же как в базе ЦРПТ. База ФИАС обновляется раз в несколько лет.
6) В UI интерфейс были добавлены кнопки:
	а) Поиск по SRID. Работа приведена к корректной. Выводит информацию по интересующим SRID.
	б) Сохранить в файл. Сохраняет на рабочий стол 2 файла с искомой информацией, по аналогии с ответом TelegramClient.
	в) Пароль тех.поддержки. Выводит пароль тех.поддержки (принимает 2 последние цифры номера РВ, либо номер целиком).
	г) AKB. Выводит информацию по напряжению.
	д) MB Errors. Выводит информацию обо всех ошибках МБ за период логов (до 14 дней), включая ошибки МБ передаваемые им самим.
	е) MK Errors. Выводит информацию обо всех ошибках СБ за период логов (до 14 дней).
	ж) Спутники. Выводит информацию об ошибках модуля ГНСС за период логов (до 14 дней). Отсутствие связи со спутниками не обрабатывается.
